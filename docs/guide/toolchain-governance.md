# Toolchain Governance

Wizig toolchain governance ensures that CLI checks, host template generation, and docs requirements are all derived from a single policy source: `toolchains.toml`.

## Goals

The governance system solves four problems:

1. Version drift between CLI checks and generated templates
2. Unknown host state at scaffold time
3. Manual docs updates that get stale
4. Non-repeatable developer setup and CI outcomes

## Single Source of Truth

`toolchains.toml` is the canonical policy file. It defines:

- Schema versioning (`[schema]`)
- Doctor enforcement policy (`[doctor]`, `[doctor.tools.<id>]`)
- iOS host defaults (`[templates.ios]`)
- Android host defaults and dependency pins (`[templates.android]`)
- Docs tooling constraints (`[docs]`)

When a value changes (e.g., Gradle wrapper version), the change is made only in `toolchains.toml`, then propagated through the generation workflow.

## Data Flow

Policy values fan out through four independent consumers:

### 1. `wizig doctor`

Reads `toolchains.toml` and evaluates host tools against policy. Each tool is checked for:

- `required` — whether the tool is mandatory
- `min_version` — minimum accepted version

Reports: missing tools, unparseable version outputs, versions below minimum.

### 2. Template Generation

Consumes policy values to render host defaults:

- iOS deployment target
- Android SDK/JVM defaults
- Android version catalog pinning
- Gradle wrapper properties

Generated by `tools/templategen/generate_templates.py`.

### 3. Create-Time Lock File

`wizig create` writes `.wizig/toolchain.lock.json` inside every generated app, capturing:

- Policy schema version
- SHA-256 hash of exact manifest bytes used
- Host tool probe results observed at scaffold time

### 4. Development Requirements Docs

Requirements are rendered from policy by `tools/toolchains/render_docs.py`, keeping published requirements aligned with doctor/template policy.

## Strict Mode

`wizig doctor` supports strict mode:

| Priority | Source |
|----------|--------|
| 1 | `--strict` or `--no-strict` CLI flag |
| 2 | `strict_default` in `toolchains.toml` |

- **Disabled**: doctor returns warnings for issues.
- **Enabled**: doctor exits non-zero for any issue.

## Lock File Lifecycle

Lock files are generated during `wizig create` at `<app>/.wizig/toolchain.lock.json`.

Use cases:

- Confirm which policy snapshot was used
- Confirm detected host versions at creation time
- Compare local and CI environments during debugging

Existing lock files remain historical evidence unless explicitly regenerated.

## Updating Policy

When changing `toolchains.toml`:

```sh
# 1. Edit toolchains.toml
# 2. Regenerate templates
python3 tools/templategen/generate_templates.py --out build/generated/templates

# 3. Regenerate docs
python3 tools/toolchains/render_docs.py

# 4. Build and test
zig build test
zig build
```

### Freshness Checks

```sh
python3 tools/toolchains/render_docs.py --check
```

## CI Recommendations

Use governance checks as explicit CI gates:

1. `zig build test`
2. `zig build`
3. `python3 tools/toolchains/render_docs.py --check`
4. (Optional) `wizig doctor --strict --sdk-root <repo_root>`

## Common Failure Modes

| Symptom | Cause | Fix |
|---------|-------|-----|
| Doctor reports missing `toolchains.toml` | Incorrect `--sdk-root` or incomplete SDK layout | Use explicit root containing `toolchains.toml` |
| Template versions don't match manifest | Templates not regenerated after manifest change | Re-run templategen |
| Docs show stale requirements | `render_docs.py` not run after policy update | Re-run docs renderer |
| Lock file versions look unexpected | Tool version parser mismatch | Re-run `wizig doctor`, update probe heuristics |

See [Toolchain Manifest Reference](../reference/toolchain-manifest.md) for field-level documentation.
