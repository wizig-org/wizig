# Code Style & Conventions

## Zig Conventions

### Allocator Patterns

- Use arena allocators throughout command execution for predictable memory management.
- Pass allocators explicitly — never use global state.
- Prefer `ArenaAllocator` for command-scoped work that can be bulk-freed.

### Error Handling

- Use Zig error unions (`!T`) for fallible operations.
- Define domain-specific error sets rather than using `anyerror`.
- Let errors propagate naturally with `try` — catch only at boundaries.

### Naming

- Functions: `camelCase` (Zig convention)
- Types: `PascalCase`
- Constants: `snake_case` or `SCREAMING_SNAKE_CASE` for compile-time constants
- Files: `snake_case.zig`

### Source Documentation

- Use `//!` at file top for module-level context.
- Use `///` on public declarations to document behavior and invariants.
- Focus on contracts and constraints, not obvious syntax.

## FFI Conventions

- All exported FFI functions use `export fn` with C ABI compatible signatures.
- Return integer status codes, never Zig errors across the ABI boundary.
- Use thread-local error envelopes for structured error details.
- Prefix all exported symbols with `wizig_runtime_` or `wizig_ffi_`.

## Generated Code

- Treat all files under `.wizig/generated/` as build artifacts.
- Never hand-edit generated files.
- Generated code should be deterministic — same inputs produce identical outputs.

## Testing

- Tests are standard Zig inline `test` blocks.
- Place tests in the same file as the code they test.
- Test names should describe the behavior being verified.
- Use `std.testing.expect*` functions for assertions.

## Project Conventions

- **Arena allocators** for command execution — bulk-free at command completion.
- **Toolchain governance** — all version requirements flow from `toolchains.toml`.
- **SDK vendoring** — projects are self-contained via `.wizig/` directory.
- **Template generation** — host project templates are generated by Python scripts, not hand-maintained.
- **Deterministic output** — codegen, lockfiles, and docs must produce identical output for identical inputs.
